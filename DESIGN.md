# 디자인

## 단일 기준
- 상태: 활성
- 마지막 갱신: 2026-05-21
- 주요 제품 화면: macOS 로그 작업공간, 소스 사이드바, 로그 목록, 상세 검사기, 로컬 파일/작업공간 열기, 최근 통합 로그 가져오기, 진단 내보내기.
- 검토한 근거: `README.md`, `docs/architecture.md`, `docs/roadmap.md`, `Sources/LogdeckCore/Views/ContentView.swift`, `Sources/LogdeckCore/Views/SourceSidebarView.swift`, `Sources/LogdeckCore/Views/LogTableView.swift`, `Sources/LogdeckCore/Views/LogInspectorView.swift`, `Sources/LogdeckCore/ViewModels/LogWorkspaceViewModel.swift`.

## 브랜드
- 성격: 차분하고, 로컬 우선이며, 기술적이고, 정확하고, 빠르다.
- 신뢰 신호: 텔레메트리 없음, 로컬 파일 처리, 명확한 상태 메시지, 되돌릴 수 있는 필터, 눈에 보이는 진단 내보내기.
- 피할 것: 마케팅형 첫 화면, 장식용 그라디언트, 과도하게 큰 글자, 장난스러운 문구, 숨겨진 상태, 네트워크/클라우드 중심 표현.

## 제품 목표
- 목표: 로컬 로그 점검을 빠르고 읽기 쉽고 복구 가능하게 만든다. 사용자가 소스 선택, 소스 닫기, 이슈 확인, 원문 검토로 이동할 때 맥락을 잃지 않게 한다.
- 비목표: 클라우드 수집, 팀 협업, 호스팅 대시보드, 공개 배포 흐름, 분석 추적.
- 성공 신호: 첫 실행 행동이 명확하다, 활성 필터가 보인다, 결과 없음 상태에서 바로 복구할 수 있다, 이슈 이동이 분명하다, 밀도 높은 로그도 훑어보기 쉽다.

## 사용자와 작업
- 주요 사용자: macOS에서 앱을 직접 실행하는 소유자/개발자.
- 사용자 작업: 하나 이상의 로그 열기, 잘못 열었거나 더 보지 않을 로그 소스 닫기, 최근 macOS 로그 확인, 검색어/심각도/메타데이터 필터링, 요청/추적 토큰 고정, 오류/장애 항목 이동, 작업공간 저장, 진단 내보내기.
- 사용 맥락: 집중 디버깅, 큰 로컬 파일, 앱 로그와 시스템 로그 혼합, 반복적인 소스 실행 개발.

## 정보 구조
- 기본 탐색: 왼쪽 소스 목록, 중앙 로그 목록, 오른쪽 검사기.
- 핵심 화면: 단일 창 SwiftUI 작업공간, 툴바, 선택적 메타데이터 필터, 로그 목록, 상태 막대, 검사기.
- 정보 위계: 빈 작업공간에서는 가져오기와 작업공간 열기만 먼저 보여야 한다. 로그가 열린 뒤에는 가져오기, 검색, 심각도 필터, 보기, 작업공간, 상세 확인 순서가 툴바에서 보여야 한다. 로그 행이 주 데이터 표면이고, 주변 로그와 원문은 선택된 로그가 있을 때만 열리는 오른쪽 검사기의 보조 상세 정보다.

## 디자인 원칙
- 원칙 1: 작업 밀도는 유지하되 숨은 상태는 모두 드러낸다.
- 원칙 2: 데이터를 좁히는 모든 동작에는 즉시 전체로 돌아가는 방법이 있어야 한다.
- 원칙 3: 오류와 장애는 강하게, 디버그와 정보는 조용하게 보여서 사용자가 이슈를 먼저 스캔하게 한다.
- 원칙 4: 표에서는 이미 컬럼으로 분리한 시간/심각도 접두사와 반복적인 앞쪽 메타데이터를 줄이고, 줄인 메타데이터와 원문은 상세 검사기에 보존한다.
- 절충: 시각적 새로움보다 네이티브 macOS 컨트롤과 로그 밀도를 우선한다. 색은 심각도, 선택, 경고 상태에만 쓴다.

## 시각 언어
- 색상: 시스템 소재와 심각도 색을 사용한다. 정보는 파랑, 경고는 주황, 오류는 빨강, 장애는 보라, 디버그는 보조 회색을 쓴다. 오류/장애/경고 외의 색은 배경 강조를 낮춰 대량 로그 스캔을 방해하지 않는다.
- 타이포그래피: 컨트롤과 메타데이터는 시스템 글꼴, 로그 본문과 카운터는 고정폭 글꼴을 쓴다.
- 간격/리듬: 8-12px 중심의 촘촘한 컨트롤 간격, 밀도 높은 행, 명령/필터/데이터/상태 영역의 명확한 구분.
- 형태/입체감: 작은 네이티브 둥근 컨트롤만 사용한다. 중첩 카드 레이아웃은 쓰지 않는다.
- 움직임: macOS 기본 전환만 사용한다. 장식 애니메이션은 쓰지 않는다.
- 아이콘: 동작과 심각도에는 SF Symbols를 쓴다. 스톡 이미지는 쓰지 않는다.

## 구성 요소
- 재사용할 기존 요소: `NavigationSplitView`, `ContentUnavailableView`, `inspector`, segmented picker, 토글 버튼, 네이티브 패널.
- 새로 바꾼 요소: 빈 작업공간에서는 시작 행동만 남기는 compact 툴바, 로그가 열린 뒤 쓰는 폭이 고정된 3단 작업 흐름형 툴바, 작업공간 열기를 포함하되 첫 화면에서는 별도 전환 메뉴를 노출하지 않는 가져오기 메뉴, 로그 열기를 주 버튼으로 승격하고 작업공간 복원은 중앙 빈 상태의 직접 행동으로 드러내며 폭이 부족하면 2행으로 접히는 시작 화면, 빈 작업공간에서 중복되는 소스 섹션을 숨기는 사이드바, 입력 위치에서 바로 복구되는 검색 지우기 버튼, 조용한 심각도 필터, 클릭하면 다음 오류/장애로 이동하는 이슈 카운터, 소스가 생긴 뒤에만 카운트와 이슈 지표를 드러내는 상태바와 사이드바 작업공간 요약, 접힌 상태에서도 의미가 보이는 메타데이터 필터 토글, 좁은 폭에서 제목/초기화와 입력칸을 2행으로 분리하는 메타데이터 필터 막대, 선택된 로그가 있을 때만 공간을 차지하고 잘림 없이 상태를 보이는 상세 패널 아이콘 토글, 접두사와 앞쪽 메타데이터 중복을 줄이되 상세 검사기에 구조화해 남기고 선택/hover 시 상세 검사기 affordance를 보여주는 키보드 탐색 가능한 커스텀 로그 그리드, 개별 해제 가능한 compact 필터 상태 막대, 심각도 배지, 선택/hover 시 닫기 affordance와 컨텍스트 메뉴를 제공하는 풍부한 소스 행, 이슈 이동을 헤더에 통합하고 주변 로그를 선택 가능한 탐색 표면으로 쓰며 원문 복사 액션을 제공하는 검사기, 표 헤더를 숨기고 로그 파일/아카이브/드래그 앤 드롭 진입점을 함께 드러내는 복구 가능한 빈 상태.
- 변형과 상태: 빈 작업공간, 필터 결과 없음, 활성 필터, 잘못된 정규식, 선택된 이슈, 고정된 상관 토큰, 비활성 실시간 추적.
- 소유 경계: UI 스타일은 SwiftUI view 파일에 둔다. 데이터/상태 요약은 `LogWorkspaceViewModel`에서 계산한다.

## 접근성
- 목표 기준: 네이티브 macOS 접근성 의미, 읽을 수 있는 레이블과 도움말.
- 키보드/포커스: 기존 단축키와 네이티브 툴바 포커스를 유지한다. 로그 그리드는 위/아래 키로 선택을 이동하고 선택 행을 보이는 위치로 스크롤해야 하며, 선택 또는 hover 행은 상세 검사기로 이어지는 표면임을 아이콘으로 보여줘야 한다.
- 상세 탐색: 검사기의 주변 로그 행은 클릭 가능한 보조 탐색 표면이어야 하며, hover와 선택 상태가 구분되어야 한다.
- 대비/가독성: 심각도 색에는 반드시 텍스트 레이블을 함께 제공한다.
- 스크린 리더: 아이콘 전용 버튼에는 도움말 또는 접근성 레이블이 있어야 한다.
- 움직임 민감도: 불필요한 애니메이션과 깜빡이는 상태를 피한다.

## 반응형 동작
- 지원 범위: macOS 14 이상 데스크톱 창, 앱 최소 크기 이상.
- 레이아웃 적응: 툴바가 한 줄에 과밀해지지 않도록 보조 필터 상태를 별도 막대로 분리하되, 메타데이터 필터는 필요할 때만 펼쳐서 로그 본문 세로 공간을 보존하고 입력칸이 잘릴 폭에서는 제목/초기화 줄과 입력 줄로 분리한다. 빈 상태 시작 행동은 긴 한글 문구나 좁은 창에서 잘리지 않도록 2행으로 접힐 수 있어야 한다. 로그 본문이 먼저 읽히도록 소스/시간/심각도/검사기 고정 폭을 보수적으로 제한한다.
- 입력 방식: 포인터와 키보드 기반 macOS 사용을 우선한다.

## 상호작용 상태
- 로딩: 상태 문구가 현재 가져오기/저장 동작을 이름으로 알려야 한다.
- 빈 상태: 직접 실행 가능한 네이티브 빈 상태와 행동 버튼을 보여준다. `로그 열기`는 첫 실행의 주 버튼이어야 하고, `.logarchive`, 최근 macOS 로그, 작업공간 열기, 드래그 앤 드롭 진입점은 보조 진입점으로 함께 도달 가능해야 한다. 작업공간 복원은 숨은 메뉴에만 의존하지 않아야 한다. 선택된 로그가 없을 때 상세 검사기나 빈 표 헤더가 주 화면 공간을 빼앗지 않아야 한다. 검색, 심각도 필터, 보기 전환, 별도 작업공간 메뉴, 비어 있는 소스 섹션, 사이드바/하단 줄 수와 이슈 지표처럼 소스가 있어야 의미 있는 도구는 첫 로그를 열기 전까지 숨긴다.
- 오류: 상태 막대 또는 필터 막대에 경고/오류를 표시하고 복구 동작을 가까이 둔다.
- 활성 필터: 검색, 심각도, 메타데이터, 고정값 필터는 전체 초기화뿐 아니라 개별 해제가 가능해야 한다. 검색어는 입력 위치에서도 바로 지울 수 있어야 한다.
- 소스 관리: 열린 소스는 사이드바에서 선택/hover affordance 또는 컨텍스트 메뉴로 바로 닫을 수 있어야 한다.
- 이슈 상태: 보이는 이슈 카운터는 단순 지표가 아니라 다음 오류/장애로 이동하는 행동이어야 한다.
- 성공: 가져오기, 저장, 필터 초기화는 상태 문구로 확인시킨다. 원문 복사처럼 좁은 검사기 액션은 버튼 가까이에 짧은 완료 피드백을 함께 보여주되, 완료 피드백은 영구 상태처럼 남기지 않는다.
- 비활성: 비활성 컨트롤은 도움말이나 주변 상태로 이유가 보이게 한다.
- 재실행: 앱이 활성화되었는데 보이는 창이 없으면 주 작업공간 창을 즉시 다시 보여주고, 첫 창은 화면 밖으로 밀리지 않게 배치한다.
- 외부 파일 열기: 명령행 인자와 macOS 파일 열기 이벤트가 같은 URL을 동시에 전달해도 소스는 한 번만 추가한다.
- 오프라인/느린 네트워크: 해당 없음. 앱은 로컬 우선이다.

## 문구 톤
- 톤: 짧고 직접적이며 기술적이다.
- 용어: "소스", "작업공간", "타임라인", "이슈", "고정값", "메타데이터", "진단"을 일관되게 쓴다.
- 마이크로카피 규칙: 설명 문단보다 명령 이름과 상태 사실을 우선한다.
- 언어: 사용자에게 보이는 앱 문구는 한국어를 기본으로 한다. `macOS`, `.logarchive`, 파일 확장자, 원문 로그 키는 그대로 둔다.

## 구현 제약
- 프레임워크/스타일: macOS 14 대상 Swift Package의 SwiftUI.
- 디자인 토큰: 시스템 색상, 소재, SF Symbols를 쓴다. 새 의존성이나 별도 디자인 시스템 패키지는 추가하지 않는다.
- 성능: 행마다 비싼 계산을 하지 않는다. 실용적인 요약은 ViewModel에서 계산한다.
- 호환성: `swift run Logdeck` 실행 흐름과 로컬 설치 스크립트를 유지한다.
- 검증 기대: UI 변경 뒤 `swift test`, `swift build`, 로컬 앱 스모크를 실행한다.
